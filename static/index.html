from databricks.sdk import WorkspaceClient
from databricks_mcp import DatabricksMCPClient

# Initialize the Databricks Workspace client
ws_client = WorkspaceClient(
    host="databricks-host",
    token="exactly-scoped-OAuth-token",
)

# Initialize the MCP client with your app's MCP endpoint
mcp_client = DatabricksMCPClient(
    server_url='',
    workspace_client=ws_client,
)

# List available MCP tools
print(mcp_client.list_tools())

Available Tools

retrieve_memory
Hybrid retrieval (similarity + recency + importance) with near-duplicate suppression and recency tiebreak. Auto-resolves user identity. Includes trust guidance.
store_memory
Extract, classify (5 types), score, PII redact, embed, and deduplicate memories from conversation text.
list_memories
Browse stored memories for a user. Filter by type (fact / preference / experience / skill / relationship). Optional include_pruned flag for audit views.

Memory Rules — Drop Into Your Agent's System Prompt

Paste this block into your agent's system prompt so it knows when and how to call the memory tools. The most important rule is the user_id source: it must come from session metadata (the calling user's email), not from what the user types in chat. 

MEMORY RULES (follow these exactly):

USER IDENTIFICATION — most important rule:
- The user_id you pass to memory tools MUST be the calling user's email
  address from session metadata (e.g. {{user.email}} if your platform
  supports template variables, or whatever identity the agent harness
  exposes for the authenticated caller).
- Do NOT use the name the user types in chat (e.g. "Alex", "Steve").
  Names users type are unstable — they may shorten, change spelling,
  or omit them entirely across sessions.
- If you genuinely cannot find an authenticated identifier, omit user_id
  rather than guessing — the server will resolve it from the auth headers
  (or fall back to "unknown" if it must).

MEMORY TOOL USAGE:
- EVERY conversation: BEFORE answering the user's first message, call
  retrieve_memory with their first message as query (and user_id from
  session metadata).
- When the user introduces themselves, shares their role, team, region,
  preferences, or any personal/professional information: call store_memory
  with the conversation text. Always store durable facts about the user
  (name, role, team, region, preferences).
- When the user asks "what do you know about me" or references previous
  sessions: call retrieve_memory.

RESPONSE RULES:
- When using retrieved memories: preface with "Based on our previous
  conversations..."
- Refer to retrieved memories in your reasoning but do not list them
  back to the user as raw data unless they ask.

Two Retrieval Modes — With or Without agent_context

retrieve_memory accepts an optional agent_context string. When provided, the server makes one batched LLM call to score how relevant each retrieved memory is to the calling agent's task, and uses that score in the hybrid ranking. When omitted, the stored general importance (set at write time) is used.

Mode How importance is determined Cost per retrieval
agent_context empty Stored general importance (write-time, agent-blind) 0 LLM calls
agent_context provided LLM-scored against the agent's task; cached per (agent_context, memory_id) 1 LLM call on first retrieval per agent; 0 on cached repeats

Why pass agent_context?

Two memories can have different importance to different agents. For example, a memory about a user's family member's name holds greater importance to an HR benefits enrollment agent than to an agent guiding a user through installing a transformer.

Recommended agent prompt pattern

Add an AGENT CONTEXT block near the top of your agent's system prompt. Tell the agent to pass that string verbatim as agent_context on every retrieve_memory call. Be explicit about what's relevant and what isn't — the LLM uses this to discriminate within memory types. 

AGENT CONTEXT (pass this verbatim as the agent_context argument
on every retrieve_memory call):

"Overtime assignment and crew routing assistant for utility supervisors.
Highly relevant: the user's role, region, crew/team assignments, manager,
direct reports, shift and overtime preferences, and policies they have
previously asked about. Less relevant: family, hobbies, or unrelated
personal interests."

Tips for writing a good agent_context

  • Name the agent's task. "Overtime routing assistant" beats "general assistant."
  • Call out what's relevant and irrelevant. Both directions help the scorer discriminate within a memory type.
  • Keep it stable. The cache key is the exact agent_context string — minor wording changes invalidate the cache.
  • Don't smuggle the user query into context. Context describes the agent; the query argument describes the question.

Memory Types

On store_memory, the server extracts atomic memories from the conversation and classifies each into one of five types. Typing prevents false dedup (e.g. "uses Hadoop" as a fact vs. "wants to move off Hadoop" as a preference are different memories even though they're semantically similar) and enables type-filtered retrieval via list_memories.

Type What it captures Examples
fact Objective information about the user or their organization "Works at PPL Corp", "Region is Southeast"
preference Stated or implied preferences and habits "Prefers concise responses", "Always pays crews double time on Fridays"
experience Past experiences or current situation "Demoed Lakebase branching on April 15", "Has been on call for 3 weeks"
skill Technical skills, training, or expertise "Knows Spark Structured Streaming", "Certified in OSHA crane operation"
relationship People, teams, or organizational relationships "Manager is Steve Phillips", "Has a spouse and child"

Features

User identityAuto-resolved from authenticated session (falls back to user_id parameter)
Memory typesfact, preference, experience, skill, relationship
PII redactionRegex-based (phone, email, SSN, credit card)
DeduplicationCosine similarity threshold (0.92)
RetrievalHybrid scoring: 50% similarity + 20% recency + 30% importance, with near-duplicate suppression and recency tiebreak
Soft deletepruned_at column; retrieval filters out pruned rows. Background pruning job runs weekly.
Trust guidanceEvery retrieval includes a trust note for the calling agent
BackendLakebase (serverless Postgres) + pgvector with HNSW index
